Governance Gaps — Things the Spec Doesn't Cover Yet

These are not components — they are process failures
The Component Lifecycle doc is excellent. But several governance mechanisms exist only as intentions, not enforced processes.

No component request SLA

The proposal process (Linear/Jira ticket) is defined. But there is no SLA on how quickly a request is reviewed or rejected. Clients and internal teams are left waiting with no timeline.

Fix: Define a 5-business-day review SLA. Auto-reject after 14 days of inactivity.

Deprecation is undated

The deprecation protocol specifies the process (red tag, console.warn, remove in next major) but no calendar-based timeline. Deprecated components linger indefinitely because there is no enforcement date.

Fix: Require a specific removal date (e.g., "removed in v4.0.0, due Q3 2026") in all deprecation notices.

No breaking change notification channel

When a Major version ships, how do existing clients learn about breaking changes? There is no defined channel — no changelog email, no status page, no Slack bot. Each client discovers breaks independently.

Fix: Create a chiti-changelog mailing list. Auto-publish to it on every major/minor release via CI.

Ethical review has no enforcement mechanism

The Ethical Protocol lists what is banned (scarcity timers, forced subscriptions). But there is no audit step in the component lifecycle that checks for violations before merge. It is a pledge, not a gate.

Fix: Add an "Ethics Review" checkbox to every Component Request ticket. Require sign-off before Figma design begins.
